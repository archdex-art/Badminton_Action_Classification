1 · Executive Summary

This project converts an academic proof-of-concept — classifying four badminton shots from AlphaPose skeletons with a 5-layer LSTM (reported 80% accuracy on 400 clips) — into a modular, tested, deployable system. The original idea is sound: skeleton keypoints are a compact, privacy-preserving, lighting-invariant representation of the biomechanics that define a shot. The original engineering, however, would not survive production.

Three changes were decisive, and all three are implemented and tested:

  1. Keypoint normalization. The paper fed raw pixel coordinates, so the model keyed on where the player stood in frame rather than how the body moved. We root-center on the pelvis and scale by torso length, producing translation- and scale-invariant features (proven by unit tests).
  2. Leakage-safe evaluation. Splitting frames/sequences risks placing clips from the same video on both sides of the split, inflating accuracy. We enforce video/player-disjoint splits.
  3. Right-sized model + honest metrics. A 2-layer BiLSTM with temporal attention replaces the over-parameterized 5×128 stack; reporting is per-class F1 + calibration (ECE), not a single noisy accuracy point.
Bottom line: the architecture is production-ready and validated end-to-end. The binding constraint on real-world accuracy is data volume (400 clips), not the system. Expanding the dataset is the highest-leverage next investment.

2 · Source Paper Analysis

The paper (Badminton Action Classification Based on Human Skeleton Data Extracted by AlphaPose) pursues shot classification without shuttlecock detection, using only the player's skeleton.

DimensionAs described in the paper
Task4-class shot recognition: backhand drive, backhand net shot, forehand clear, forehand drive
Pose sourceAlphaPose (RMPE: STN + SPPE + Pose-NMS), 17 COCO keypoints
FeaturesPer joint (x, y, confidence) → input dimension 51, raw pixels
DatasetKaggle; 100 videos/class (400 total), 10 frames/video = 4,000 frames; 8:1:1 split
Frame selectionFrame Difference Method (inter-frame pixel difference)
Model5 stacked LSTM layers, 128 hidden units, FC head; Adam (lr 1e-3), cross-entropy, 110 epochs
ResultsLSTM 80% vs. CNN 60% test accuracy

What the paper gets right

  • Skeleton-first, shuttle-free — ~10× lower dimensionality than raw video; robust to background, lighting, and apparel.
  • LSTM > frame-CNN — shot identity lives in the temporal swing trajectory; a frame classifier discards it. The conclusion is correct.
  • Frame-difference keyframing — a reasonable, cheap motion heuristic.

Feature engineering — the highest-leverage change

Raw keypoints [T, 17, 3] are transformed into a camera-invariant, kinematic vector: (1) translation invariance via pelvis-centering; (2) scale invariance via torso length; (3) optional shoulder-line rotation alignment; (4) joint velocities and key bone angles (elbow, shoulder, knee); (5) confidence-masked interpolation with a per-joint validity flag.

Real-data pipeline (RTMPose) & cross-validation

The full chain was exercised on the real Kaggle dataset (data/archive: 5 classes, 662 clips at 1080p / 20 fps). A pluggable RTMPose backend (rtmlib + ONNX, no mmcv build) decodes each clip, runs detection + top-down pose, and keeps the most-confident person per frame → [T, 17, 3] keypoints. Because player identities are unknown, evaluation uses video-disjoint splits and a group-stratified k-fold runner that reports mean ± std rather than a single point.

Starting baseline (662 clips, 5 classes, 16-frame, 5-fold video-disjoint CV): accuracy 0.456 ± 0.035, macro-F1 0.443 ± 0.044, ECE 0.064 ± 0.016 — later lifted to 0.590 by whole-clip sampling (see below). This was the honest baseline: ~46% on 5 classes (chance = 20%), well-calibrated, and stable across folds. Macro-F1 ≈ accuracy confirms the model learns all five classes rather than collapsing to a majority. It sits well below the paper's reported 80% — but that figure was 4 classes on a frame-level split (susceptible to same-clip leakage); this is 5 classes under strict video-disjoint evaluation. The gap is the cost of measuring honestly.

For reference, a 20-clip/class subset (~100 clips) scored only 0.29 ± 0.09 — near chance — demonstrating that the leakage-safe evaluation refuses to manufacture accuracy from too little data, and that data volume is the binding constraint.

Where the remaining accuracy is

The highest-leverage improvements, in order: (1) more frames per clip — done (whole-clip sampling, +13 pts CV); (2) ST-GCN to exploit the skeletal graph — implemented (configs/stgcn.yaml, 411k params vs the BiLSTM's 687k), benchmarked at 0.502 ± 0.059 CV; (3) hyper-parameter search (hidden size, sequence length, learning rate); (4) a larger, player-labelled corpus enabling player-disjoint evaluation.

ST-GCN: structure-aware model

The BiLSTM flattens all 17 joints into one vector, discarding which joints are physically connected. ST-GCN (Yan et al. 2018) instead convolves over the COCO-17 skeleton graph in space and over time, with learnable per-edge importance. It is wired behind the same model factory: setting model.name: stgcn auto-switches the feature extractor to a joint-grouped graph layout (5 channels/joint: x, y, validity, vₓ, v_y), so the dataset, training, CV, and serving code are unchanged. A compact 4-block stack keeps it leaner than the BiLSTM to suit the ~460-clip training set. Real-data 5-fold CV = 0.502 ± 0.059 — below the BiLSTM's 0.590, as expected when data is scarce (graph models need more). It should overtake the BiLSTM once the dataset grows.

Production model & GPU pipeline

Improvement (1) was implemented: a whole-clip uniform sampler (read_video_uniform) now spans the entire clip with 24 frames, and the full pipeline runs on a CUDA GPU via the bundled Colab notebook (docs/colab_pipeline.ipynb) — RTMPose extraction that takes ~2 hrs on CPU finishes in minutes. The deployed checkpoint (model.pt) is the whole-clip, calibrated (temperature 1.71) BiLSTM over all 5 classes.

Independent end-to-end verification (local, real video → RTMPose → deployed model.pt, 10 held clips): 5/10 correct — consistent with the model's accuracy band. Calibration and abstention behaved correctly (low-confidence predictions abstained; the single high-confidence call, 0.87, was correct), and the misclassifications were the expected drive/clear/net-shot confusions among visually similar fast strokes. This confirms the trained model is servable and behaves sanely outside the training harness.

Two firm, apples-to-apples conclusions:
  • Whole-clip coverage is a large win: +13.4 pts. BiLSTM 5-fold CV rises 0.456 → 0.590 ± 0.053 (macro-F1 0.443 → 0.575) just by sampling the whole stroke instead of the clip's first ~1.6 s. This is the single biggest lever found, and confirms the report's core diagnosis: accuracy was limited by what the features captured (the swing was being missed), not the architecture.
  • BiLSTM beats ST-GCN (0.590 vs 0.502). The graph model underperforms by ~9 points — expected, since ST-GCN is data-hungry and the training set is only ~460 clips. The structure-aware model will likely overtake the BiLSTM only once the dataset grows.
  • Best model: BiLSTM whole-clip, 0.590 ± 0.053. The next lever is now clearly data volume (transfer learning / external datasets), not a fancier architecture.

12 · Appendix — How to Run

make dev      # install package + dev tooling
make test     # 16 tests, coverage report
make data     # generate synthetic dataset (no AlphaPose needed)
make train    # train -> calibrate -> checkpoint (model.pt)
make serve    # FastAPI service on :8000

# Predict from pre-extracted keypoints:
curl -X POST localhost:8000/v1/predict/keypoints \
     -H 'Content-Type: application/json' \
     -d '{"keypoints": [[[x,y,conf], ... x17], ... xT]}'

The Makefile auto-detects the Python interpreter (python3 or python); override with make train PYTHON=/path/to/python.
